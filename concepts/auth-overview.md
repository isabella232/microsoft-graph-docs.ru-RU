# <a name="get-access-tokens-to-call-microsoft-graph"></a>Получение маркеров доступа для вызова Microsoft Graph

Чтобы приложение могло вызывать Microsoft Graph, оно должно получить маркер доступа из Azure Active Directory (Azure AD) — облачной службы идентификации от корпорации Майкрософт. Маркер доступа содержит сведения (или утверждения) о приложении и его разрешениях на доступ к ресурсам и API, доступным через Microsoft Graph. Чтобы получить маркер доступа, приложение должно иметь возможность пройти проверку подлинности в Azure AD, а также получить от пользователя или администратора разрешение на доступ к необходимым ему ресурсам Microsoft Graph. 

В этой статье представлены общие сведения о маркерах доступа, Azure AD и способах получения маркеров доступа приложением. Если вы уже знаете, как интегрировать приложение с Azure AD для получения маркеров, то можете переходить к разделу [Дальнейшие действия](#next-steps), в котором представлены сведения и примеры, относящиеся непосредственно к Microsoft Graph. 

> **Важно!**  В настоящее время мы вносим изменения в порядок применения политик условного доступа к Microsoft Graph. Вам необходимо обновить свои приложения, чтобы они могли обрабатывать сценарии, в которых выполняется настройка политик условного доступа. Дополнительные сведения и рекомендации см. в статье [Руководство для разработчиков по условному доступу в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/develop/active-directory-conditional-access-developer).

## <a name="what-is-an-access-token-and-how-do-i-use-it"></a>Что такое маркер доступа и как его использовать?

Маркеры доступа, выдаваемые службой Azure AD, представляют собой маркеры JSON Web Token (JWT) с кодировкой Base 64. Они содержат сведения (утверждения), с помощью которых веб-API, защищенные службой Azure AD, например Microsoft Graph, проверяют вызывающую сторону и гарантируют наличие у нее надлежащих разрешений на выполнение запрашиваемой операции. При вызове Microsoft Graph маркеры доступа можно рассматривать как непрозрачные. Маркер доступа всегда следует передавать по защищенному каналу, например по протоколу TLS (HTTPS).

## <a name="what-are-microsoft-graph-permissions"></a>Что такое разрешения Microsoft Graph?
Набор Microsoft Graph предоставляет широкий выбор детализированных разрешений для контролируемых им ресурсов. Эти разрешения представлены в виде строк и предоставляют приложениям доступ к ресурсам Microsoft Graph, таким как пользователи, группы, почта пользователей. Например:

- разрешение _User.Read_ позволяет приложению просматривать профиль вошедшего пользователя;
- разрешение _Mail.Send_ позволяет приложению отправлять почту от имени вошедшего пользователя.

Разрешения делятся на два типа.

- Делегированные разрешения используются приложениями с вошедшим пользователем. Привилегии пользователя делегируются приложению, которое вызывает Microsoft Graph от имени пользователя. Многие из этих разрешений может предоставлять пользователь, но для некоторых требуется согласие администратора.  
- Разрешения приложений используются приложениями, работающими без пользователя. Они часто предоставляют приложению высокий уровень привилегий в организации и всегда требуют согласия администратора.

Полный список разрешений Microsoft Graph, а также описание различий между разрешениями приложения и делегированными разрешениями см. в [справочнике по разрешениям](permissions-reference.md).

## <a name="where-does-my-app-get-an-access-token"></a>Где приложение может получить маркер доступа?
Приложение получает маркеры доступа из Azure Active Directory (Azure AD) — облачной службы идентификации от корпорации Майкрософт. Чтобы получить маркер доступа, приложение обменивается HTTP-запросами и откликами с Azure AD, используя стандартные отраслевые протоколы, определенные в спецификациях OAuth 2.0 и OpenID Connect 1.0. В этих протоколах описываются конечные точки Azure AD и способы обмена с ними (потоки проверки подлинности), используемые приложением для безопасной проверки подлинности в Azure AD и получения маркеров доступа.  

На очень простом уровне, чтобы получить маркер доступа, приложение обменивается HTTP-запросами со следующими конечными точками:

- конечной точкой `/authorize`, в которую приложение может направить пользователя для проверки подлинности в Azure AD и предоставления необходимых приложению разрешений;
- конечной точкой `/token`, где приложение может получить маркер доступа, когда пользователь даст согласие.

Примечание. Это нестрогие определения. В зависимости от используемого приложением протокола, оно может получать маркеры доступа непосредственно из конечной точки `/authorize` или проходить проверку подлинности напрямую в конечной точке `/token`. 

Ниже представлен пример одного набора конечных точек `/authorize` и `/token`, предоставляемого службой Azure AD версии 2.0.

```
https://login.microsoftonline.com/common/oauth2/v2.0/authorize
https://login.microsoftonline.com/common/oauth2/v2.0/token

```
Azure AD предоставляет два набора конечных точек: Azure AD и Azure AD версии 2.0. Основное отличие между ними состоит в том, что конечная точка Azure AD поддерживает только рабочие и учебные учетные записи (т. е. связанные с клиентом Azure AD), в то время как Azure AD версии 2.0 также поддерживает учетные записи Майкрософт, такие как учетные записи _Live.com_ и _outlook.com_. Это означает, что если используется конечная точка Azure AD, приложение подходит только для организаций, а если применяется Azure AD версии 2.0, то оно подходит как для потребителей, так и для организаций. 

Маркеры из конечных точек Azure AD и Azure AD версии 2.0 не являются взаимозаменяемыми, поэтому прежде чем приступать к работе над приложением для рабочей среды, следует выбрать одну из них. Так как конечная точка Azure AD версии 2.0 была выпущена позже и к ней все еще добавляются функции, следует учитывать некоторые важные ограничения, принимая решение о том, какую конечную точку использовать для приложения в рабочей среде. Дополнительные сведения см. в разделе [Выбор между конечными точками Azure AD и Azure AD версии 2.0](#deciding-between-the-azure-ad-and-azure-ad-v20-endpoints).

## <a name="whats-the-difference-between-oauth-20-and-openid-connect"></a>В чем разница между OAuth 2.0 и OpenID Connect?

OAuth 2.0 — это протокол авторизации. Он определяет, как приложение может получать маркеры доступа, проходя проверку подлинности непосредственно в Azure AD или перенаправляя пользователя, чтобы он прошел проверку подлинности в Azure AD и согласился предоставить разрешения, запрашиваемые приложением. В первом случае приложение получает маркер доступа, с помощью которого оно может вызывать Microsoft Graph от своего имени. Во втором случае приложение получает маркер доступа, с помощью которого оно может вызывать Microsoft Graph от имени пользователя. Однако при использовании протокола OAuth 2.0 приложение не получает никаких сведений о пользователе и о том, как он прошел проверку подлинности Azure AD. Потоки OAuth 2.0 чаще всего используются мобильными и нативными приложениями, которым уже известны личные данные пользователя, либо такими приложениями, как фоновые службы и управляющие программы, которые вызывают Microsoft Graph от своего имени, а не от имени пользователя.

OpenID Connect расширяет протокол OAuth 2.0, добавляя уровень удостоверений. При использовании OpenID Connect, помимо маркера доступа, приложение также может получить маркер идентификатора из службы Azure AD. Маркеры идентификаторов OpenID Connect содержат утверждения об удостоверении пользователя и сведения о том, как и где он прошел проверку подлинности. Потоки OpenID Connect обычно используются веб-приложениями, в том числе одностраничными (SPA). С помощью маркера идентификатора эти приложения могут настраивать свое поведение для пользователя, для которого запрашивался маркер доступа. Кроме того, во многих случаях вход пользователей поручается службе Azure AD. При этом становятся доступны такие возможности, как единый вход.

## <a name="what-kind-of-apps-can-i-call-microsoft-graph-from"></a>Из каких приложений можно вызывать Microsoft Graph?
Вы можете вызывать Microsoft Graph из приложений следующих типов: 

- **Нативные приложения**. Приложения, работающие на таких устройствах, как настольные компьютеры, планшеты и мобильные телефоны. Эти приложения используют собственную операционную систему (ОС) устройства (например, iOS, Android или Windows) для представления пользователей и вызова Microsoft Graph от имени пользователя.
- **Веб-приложения**. Приложения, которые работают на сервере и взаимодействуют с вошедшим пользователем через агента пользователя (как правило, веб-браузер). Большая часть уровня представления обрабатывается на сервере, а вызовы Microsoft Graph совершаются со стороны сервера от имени пользователя.
- **Одностраничные приложения (SPA)**. Веб-приложения с полнофункциональными пользовательскими интерфейсами, которые обрабатывают большую часть уровня представления с помощью сценариев на стороне клиента в браузере. Вызовы Microsoft Graph совершаются из клиентского сценария с использованием таких технологий, как AJAX, и таких платформ, как Angular.js. Вызовы совершаются от имени пользователя.
- **Фоновые службы и управляющие программы**. Фоновые службы и управляющие программы, которые работают на сервере в случаях, когда вход пользователя не предусмотрен, и вызывают Microsoft Graph от своего имени.
- **Веб-API**. Клиентское приложение вызывает веб-API, защищенный службой Azure AD, который затем вызывает Microsoft Graph. Эти действия выполняются от имени пользователя. Поддерживаются конечной точкой Azure AD. Поддерживаются конечной точкой Azure AD версии 2.0, только если клиент и веб-API имеют одинаковый идентификатор приложения, например собственное приложение, которое вызывает серверную часть веб-API. 

## <a name="how-do-i-get-my-app-talking-to-azure-ad-and-microsoft-graph"></a>Как обеспечить связь приложения с Azure AD и Microsoft Graph?
Чтобы получить маркер из Azure AD, приложение должно быть зарегистрировано. В случае конечной точки Azure AD версии 2.0 для регистрации приложения используется [портал Майкрософт для регистрации приложений](https://apps.dev.microsoft.com/). Для конечной точки Azure AD используется [портал Azure](https://portal.azure.com/). При регистрации приложение интегрируется с Azure AD, а также задаются координаты и идентификаторы, используемые для получения маркеров. К ним относятся:

- **Идентификатор приложения**. Уникальный идентификатор, назначенный службой Azure AD. 
- **URI/URL-адрес перенаправления**. Одна или несколько конечных точек, в которых приложение будет получать ответы от Azure AD (для нативных и мобильных приложений это URI, назначенный службой Azure AD).
- **Секрет приложения**. Пароль либо пара ключей (открытый и закрытый), используемые приложением для проверки подлинности в Azure AD. (Не требуется для нативных или мобильных приложений.)

Для приложений, использующих конечную точку Azure AD, также заранее настраиваются разрешения Microsoft Graph, необходимые приложению во время регистрации. Для веб-приложений, использующих конечную точку Azure AD версии 2.0, в некоторых случаях может потребоваться заранее настроить разрешения. 

Свойства, настроенные во время регистрации, используются в сети. Например, в приведенном ниже запросе на получение токена *client_id* — это *идентификатор приложения*, *redirect_uri* — один из зарегистрированных для приложения *URI перенаправления*, а *client_secret* — *секрет приложения*. 

```
// Line breaks for legibility only

POST /common/oauth2/v2.0/token HTTP/1.1
Host: https://login.microsoftonline.com
Content-Type: application/x-www-form-urlencoded

client_id=6731de76-14a6-49ae-97bc-6eba6914391e
&scope=user.read%20mail.read
&code=OAAABAAAAiL9Kn2Z27UubvWFPbm0gLWQJVzCTE9UkP3pSx1aXxUjq3n8b2JRLk4OxVXr...
&redirect_uri=http%3A%2F%2Flocalhost%2Fmyapp%2F
&grant_type=authorization_code
&client_secret=JqQX2PNo9bpM0uEihUPzyrh    // NOTE: Only required for web apps
```

## <a name="are-there-authentication-libraries-available"></a>Доступны ли библиотеки аутентификации?
Как и большинство разработчиков, вы наверняка будете использовать библиотеки проверки подлинности для управления взаимодействием с Azure AD. Библиотеки проверки подлинности позволяют разработчикам абстрагироваться от многих сведений о протоколе (например, о проверке, обработке файлов cookie, кэшировании маркеров и поддержке защищенных соединений) и сосредоточить внимание на разработке приложения. Корпорация Майкрософт публикует библиотеки с открытым кодом и серверное ПО промежуточного слоя как для конечной точки Azure AD, так и для конечной точки Azure AD версии 2.0. 

Для конечной точки Azure AD версии 2.0: 

- Клиентские библиотеки MSAL — библиотеки проверки подлинности от корпорации Майкрософт — доступны для платформ .NET, JavaScript, Android и Objective-c. Все платформы находятся в стадии разработки с поддержкой для рабочих сред, а в случае внесения критических изменений корпорация Майкрософт гарантирует возможность обновления.
- Серверное ПО промежуточного слоя от корпорации Майкрософт доступно для .Net Core и ASP.NET (OWIN OpenID Connect и OAuth), а также для Node.js (Microsoft Azure AD Passport.js). 
- Конечная точка версии 2.0 совместима со множеством сторонних библиотек проверки подлинности.

Полный список клиентских библиотек и серверного ПО промежуточного слоя от корпорации Майкрософт, а также совместимых сторонних библиотек см. в статье [Библиотеки проверки подлинности Azure Active Directory версии 2.0](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-libraries).

Для конечной точки Azure AD:

- Для библиотеки проверки подлинности Active Directory (ADAL) доступны клиентские библиотеки на несколько большем количестве платформ. 
- Серверное ПО промежуточного слоя доступно для .Net Core и ASP.NET, а также для Node.js. 

Полный список клиентских библиотек и серверного ПО промежуточного слоя от корпорации Майкрософт см. в статье [Библиотеки проверки подлинности Azure Active Directory](https://docs.microsoft.com/azure/active-directory/develop/active-directory-authentication-libraries).

## <a name="deciding-between-the-azure-ad-and-azure-ad-v20-endpoints"></a>Выбор между конечными точками Azure AD и Azure AD версии 2.0

Azure AD предоставляет два набора конечных точек (Azure AD и Azure AD 2.0), от которых можно получать маркеры доступа для вызова Microsoft Graph. Маркеры, полученные от разных конечных точек, не являются взаимозаменяемыми. Выбор конечных точек Azure AD не имеет критического значения для запуска примеров и изучения функций Microsoft Graph. Однако прежде чем приступать к разработке рабочего приложения, необходимо решить, какая конечная точка лучше всего подходит для сценария. Ниже представлены некоторые общие рекомендации, которые помогут вам принять решение, но наиболее актуальные и исчерпывающие сведения вы найдете в статье [Следует ли мне использовать конечную точку версии 2.0?](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-limitations) из документации по Azure Active Directory. 

Основное отличие между Azure AD и Azure AD версии 2.0 заключается в следующем:

* Azure AD поддерживает только рабочие и учебные учетные записи, то есть учетные записи, связанные с клиентом Azure AD. Это означает, что приложение может работать только в организациях.
* Azure AD версии 2.0 поддерживает как рабочие и учебные учетные записи, так и учетные записи Майкрософт, например из служб _live.com_ и _outlook.com_. Это означает, что приложение, использующее конечную точку версии 2.0, подходит как для потребителей, так и для организаций. 

Конечная точка Azure AD версии 2.0 предоставляет некоторые дополнительные преимущества. Например:

* Приложение может использовать один идентификатор приложения для нескольких платформ. Это упрощает управление приложением как для разработчиков, так и для администраторов.
* [Поддержка динамического и добавочного согласия](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-compare#incremental-and-dynamic-consent). Благодаря этой функции приложение может запрашивать дополнительные разрешения во время выполнения, совмещая запрос на согласие пользователя с требующими его функциями. Это намного удобнее для пользователей, чем необходимость соглашаться на длинный список разрешений при первом входе.  

Конечная точка Azure AD версии 2.0 была выпущена позже, чем конечная точка Azure AD, и к ней все еще добавляются функции, поэтому при выборе следует учитывать некоторые ограничения конечной точки версии 2.0. Например:

* Некоторые функции могут быть не полностью реализованы в версии 2.0. Например, приложение может не работать, если корпоративный клиент включит функции корпоративной безопасности для мобильных устройств, такие как [условный доступ с устройств](https://azure.microsoft.com/documentation/articles/active-directory-conditional-access-device-policies).
* Невозможно вызывать приложения поставщика облачных решений.
* [Встроенная проверка подлинности Windows для федеративных клиентов](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-limitations#limitations) не поддерживается. Это означает, что пользователи федеративных клиентов Azure AD не могут незаметно проходить проверку подлинности с помощью локального экземпляра Active Directory. Им потребуется повторно вводить учетные данные.

Дополнительные сведения о различиях между конечной точкой Azure AD версии 2.0 и конечной точкой Azure AD см. в статье [В чем отличия конечной точки версии 2.0?](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-compare)

>**Важно!**
>
>**Прежде чем принимать решение о том, какую конечную точку использовать при разработке приложения для рабочей среды, ознакомьтесь со статьей [Следует ли мне использовать конечную точку версии 2.0?](https://docs.microsoft.com/azure/active-directory/develop/active-directory-v2-limitations)**
