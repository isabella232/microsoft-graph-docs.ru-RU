## <a name="ways-to-contribute"></a>Как внести свой вклад

Вы можете внести свой вклад в [документацию по Microsoft Graph](http://developer.microsoft.com/graph/docs) следующими способами:

- дополнять статьи в [общедоступном репозитории документов для разработчиков Microsoft Graph](https://github.com/microsoftgraph/microsoft-graph-docs);
- сообщать об ошибках в документации в [разделе Issues на сайте GitHub](https://github.com/microsoftgraph/microsoft-graph-docs/issues);
- Добавляйте запросы документации на [странице UserVoice для платформы разработчиков Office](http://officespdev.uservoice.com).

## <a name="before-we-can-accept-your-pull-request"></a>Необходимые условия для принятия запроса на включение внесенных изменений

### <a name="minor-corrections"></a>Небольшие исправления

Для внесения небольших исправлений и уточнений в документацию и примеры кода в этом репозитории не требуется лицензионное соглашение об участии (CLA). Изменения принимаются в виде запросов на включение внесенных изменений. Мы будем делать все возможное, чтобы рассматривать эти запросы в течение 10 рабочих дней.

### <a name="larger-submissions"></a>Значительные изменения

Чтобы принять ваш запрос на вытягивание новых документов и примеров кода или внесение значительных изменений в них, необходимо подписать лицензионное соглашение с участником (CLA), если вы относитесь к одной из следующих групп:

- участники группы Microsoft Open Technologies;
- участники, не являющиеся сотрудниками Майкрософт.

Как участник сообщества, **вы должны подписать лицензионное соглашение с участником (CLA), прежде чем сможете вносить большие изменения в этот проект**, но заполнить и отправить документацию достаточно один раз. Внимательно изучите документ. Кроме того, может потребоваться, чтобы ваш работодатель подписал его.

Подписывая лицензионное соглашение с участником (CLA), вы не получаете прав на внесение изменений в основной репозиторий, но участники групп разработчиков Майкрософт смогут просматривать и оценивать ваш вклад, а также будут указывать ваше авторство.

Вы можете скачать лицензионное соглашение с участником (CLA) или подписать его цифровой подписью [здесь](https://cla.microsoft.com). Когда мы получим и обработаем ваше соглашение CLA, мы будем стараться рассматривать ваши запросы на вытягивание в течение 10 рабочих дней.

## <a name="use-github-git-and-this-repository"></a>Использование GitHub, Git и этого репозитория

**Примечание.** Большую часть сведений из этого раздела можно найти в [справочных статьях GitHub][].  Если вы знакомы с Git и GitHub, переходите к разделу **Добавление и редактирование содержимого**, в котором описано, как работать над кодом и содержимым в этом репозитории.

### <a name="setting-up-your-fork-of-the-repository"></a>Настройка вашего ответвления в репозитории

1. Чтобы добавлять информацию в этот проект, настройте учетную запись GitHub. Если вы еще не сделали это, перейдите на [домашнюю страницу GitHub][] и выполните необходимые действия.
2. Настройте Git на своем компьютере. Выполните действия, указанные в [руководстве по настройке Git][Set Up Git].
3. Создайте свое ответвление этого репозитория. Для этого нажмите кнопку **Fork** (Развилка) в верхней части страницы.
4. Скопируйте свое ответвление на локальный компьютер. Для этого откройте GitBash. Введите в командной строке следующую команду:

```cmd
git clone https://github.com/{your user name}/microsoft-graph-docs.git
```

Затем создайте ссылку на корневой репозиторий с помощью следующих команд:

```cmd
cd microsoft-graph-docs
git remote add upstream https://github.com/microsoftgraph/microsoft-graph-docs.git
git fetch upstream
```

Поздравляем! Вы настроили свой репозиторий. Вам не потребуется повторять эти действия.

### <a name="contribute-and-edit-content"></a>Добавление и редактирование содержимого

Чтобы добавлять и редактировать содержимое было удобнее, следуйте указанной ниже процедуре.

1. Создайте новую ветвь.
1. Добавьте или отредактируйте содержимое.
1. Отправьте запрос на включение внесенных изменений в основной репозиторий.
1. Удалите ветвь.

Ограничьте каждую ветвь одним понятием или одной статьей, чтобы упростить работу и снизить риск возникновения конфликтов при объединении. Для новой ветви подходят следующие типы дополнений:

- новая статья (и соответствующие изображения);
- исправления орфографии и грамматики в статье;
- применение одного изменения форматирования для большого количества статей (например, применение нового нижнего колонтитула для авторских прав).

#### <a name="create-a-new-branch"></a>Создание ветви

1. Откройте GitBash.
1. Введите в командной строке команду `git pull upstream master:<new branch name>`. При этом создается локальная ветвь, которая копируется из последней главной ветви *microsoftgraph*. **Примечание.** Для внесения внутренних изменений участникам необходимо заменить идентификатор `master` в команде на нужную дату публикации.
1. Введите в командной строке команду `git push origin <new branch name>`. Так вы сообщите GitHub о новой ветви. Новая ветвь появится в вашем ответвлении репозитория на сайте GitHub.
1. Введите команду `git checkout <new branch name>`, чтобы переключиться на новую ветвь.

#### <a name="add-new-content-or-edit-existing-content"></a>Добавление или редактирование содержимого

Чтобы редактировать файлы, открывайте их в любом редакторе и изменяйте. Чтобы создать новый файл, откройте любой редактор и сохраните новый файл в подходящем расположении в локальной копии репозитория. Не забывайте часто сохранять файлы во время работы.

Файлы на локальном компьютере — это рабочая копия новой ветви, которую вы создали в локальном репозитории. Изменение файлов в этой папке не влияет на локальный репозиторий, пока вы не сохраните изменение. Чтобы сохранить изменение в локальном репозитории, введите следующие команды в GitBash:

```cmd
git add .
git commit -v -a -m "<Describe the changes made in this commit>"
```

Команда `add` добавляет изменения в область промежуточного хранения для подготовки к сохранению в репозитории. Точка после команды `add` указывает, что вы хотите поместить на промежуточное хранение все добавленные и измененные файлы, рекурсивно проверяя вложенные папки. (Если вы не хотите сохранять все изменения, вы можете добавить определенные файлы. Вы также можете отменить изменения. Для получения справки введите команду `git add -help` или `git status`.)

Команда `commit` применяет промежуточные изменения в репозитории. Параметр `-m` указывает, что вы предоставляете в командной строке комментарий к изменению. Если у вас нет конкретных сроков публикации, вы можете написать "опубликовать как можно скорее".  Параметры -v и -a указывать необязательно. Параметр -v включает подробные выходные данные команды, а параметр -a выполняет те же действия, что и уже запущенная нами команда add.)

Вы можете сохранять изменения несколько раз во время работы или только один раз, когда все будет готово.

#### <a name="submit-a-pull-request-to-the-main-repository"></a>Отправка запроса на включение внесенных изменений в основной репозиторий

Кода вы завершите работу и будете готовы объединить ее с центральным репозиторием, выполните указанные ниже действия.

1. Введите в командной строке GitBash команду `git push origin <new branch name>`. `origin` — это репозиторий GitHub, с которого вы скопировали свой локальный репозиторий. Эта команда передает текущее состояние новой ветви, в том числе все изменения, сохраненные на предыдущих шагах, в вашем ответвлении GitHub.
2. На сайте GitHub перейдите к новой ветви в своем ответвлении.
3. Нажмите кнопку **Pull Request** (Запрос на включение внесенных изменений) в верхней части страницы.
4. Убедитесь, что в качестве базовой ветви указана ветвь `microsoftgraph/microsoft-graph-docs@master`, а в качестве главной — `<your username>/microsoft-graph-docs@<branch name>`.
5. Нажмите кнопку **Update Commit Range** (Обновить диапазон применения).
6. Укажите название запроса и опишите изменения, которые вы вносите. Если вы исправили ошибку в элементе UserVoice или GitHub, обязательно сообщите об этом в описании.
7. Отправьте запрос на включение внесенных изменений.

Его обработает один из администраторов сайта. Ваш запрос появится в репозитории microsoftgraph/microsoft-graph-docs site в разделе Issues (Проблемы). Когда запрос будет принят, проблема будет устранена.

#### <a name="create-a-new-branch-after-merge"></a>Создание ветви после объединения

После успешного объединения (т. е. принятия запроса на включение внесенных изменений) не продолжайте работать с локальной ветвью, которая была успешно добавлена в репозиторий. Если вы отправите еще один запрос, это может привести к конфликтам при объединении. Вместо этого, если вы хотите внести еще одно изменение, создайте новую локальную ветвь на основе успешно объединенной ветви.

Допустим, локальная ветвь X была успешно объединена с главной ветвью microsoftgraph/microsoft-graph-docs, а вы хотите внести дополнительные изменения в объединенное содержимое. Создайте локальную ветвь X2, на основе главной ветви microsoftgraph/microsoft-graph-docs. Для этого откройте GitBash и выполните следующие команды:

```cmd
cd microsoft-graph-docs
git pull upstream master:X2
git push origin X2
```

Теперь у вас есть локальные копии (в новой локальной ветви) изменений, внесенных в ветви X. Кроме того, ветвь X2 содержит все изменения, внесенные другими авторами, так что если ваша работа зависит от чужих материалов (например, общих изображений), они будут доступны в новой ветви. Вы можете убедиться, что ваши предыдущие изменения (и чужие материалы) включены в новую ветвь, открыв ее...

```cmd
git checkout X2
```

…и проверив ее содержимое. Команда `checkout` обновляет все файлы на диске до текущего состояния ветви X2.
Получив новую ветвь для изменения, вы можете вносить изменения в содержимое и сохранять их в обычном порядке.
Однако во избежание работы в объединенной ветви (X) рекомендуем удалить ее (см. раздел **Удаление ветви** ниже).

#### <a name="delete-a-branch"></a>Удаление ветви

Успешно объединив свои изменения с центральным репозиторием, вы можете удалить соответствующую ветвь, так как она больше не требуется. Для дополнительной работы требуется новая ветвь.

Чтобы удалить ветвь, сделайте следующее:

1. Введите в командной строке GitBash команду `git checkout master`.  Это гарантирует, что вы не работаете в удаляемой ветви (это не допускается).
2. Затем введите в командной строке команду `git branch -d <branch name>`.  При этом ветвь будет удалена только на локальном компьютере, если она была успешно объединена с репозиторием верхнего уровня. (Вы можете переопределить это правило с помощью флага `–D`, но для начала убедитесь, что это необходимо.)
3. Наконец, введите в командной строке команду `git push origin :<branch name>` (с пробелом перед двоеточием и без пробела после него).  При этом ветвь будет удалена из вашего ответвления GitHub.

Поздравляем! Вы успешно внесли свой вклад в проект.

## <a name="how-to-use-markdown-to-format-your-topic"></a>Как форматировать статью с помощью Markdown

### <a name="standard-markdown"></a>Стандартные параметры Markdown

Во всех статьях в этом репозитории используется Markdown.  Полное введение (и полное описание синтаксиса) можно найти на [домашней странице Markdown] [], а здесь мы рассмотрим лишь основные моменты.

Если вам необходим хороший редактор, попробуйте [Visual Studio Code][vscode].

### <a name="markdown-basics"></a>Основы Markdown

Ниже представлен список наиболее распространенных операторов Markdown.

- **Разрывы строк и абзацы.** В Markdown отсутствует элемент HTML `<br />`. Вместо этого новый абзац обозначается пустой строкой между двумя блоками текста.
- **Курсив.** Элемент HTML `<i>some text</i>` записывается как `*some text*`.
- **Полужирный шрифт.** Элемент HTML `<strong>some text</strong>` записывается как `**some text**`.
- **Заголовки.** Заголовки HTML обозначаются определенным количеством символов `#` в начале строки.  Количество символов `#` соответствует уровню заголовка в иерархии (например, `#` = h1, `##` = h2 и `###` = h3).
- **Нумерованные списки.** Чтобы создать нумерованный (упорядоченный) список, начните строку с обозначения `1. `. Чтобы создать несколько элементов, вложенных в один элемент списка, отформатируйте список следующим образом:

    1. Обратите внимание, что после точки (".") ставится пробел.

       Обратите внимание также на то, что между двумя абзацами в элементе списка есть разрыв строки, а отступ здесь соответствует отступу верхней строки.

- **Маркированные списки.** Маркированные (неупорядоченные) списки почти идентичны упорядоченным, но вместо обозначения `1. ` используется символ `- `, `* ` или `+ `.  Несколько элементов списка создаются так же, как и в упорядоченных списках.
- **Ссылки.** Базовый синтаксис ссылки: `[visible link text](link url)`.
  У ссылок также могут быть сноски, которые рассматриваются в разделе **Сноски ссылок и изображений** далее в этой статье.
- **Изображения.** Базовый синтаксис изображения: `![alt text for the image](image url)`.
  У изображений также могут быть сноски, которые рассматриваются в разделе **Сноски ссылок и изображений** далее в этой статье.
- **Встроенный код HTML.** Markdown позволяет встроить код HTML, но лучше не делать этого.

### <a name="link-and-image-references"></a>Сноски для ссылок и изображений

В Markdown есть удобная функция, позволяющая пользователю вставлять сноски вместо URL-адресов для изображений и ссылок. Ниже представлен синтаксис использования этой функции.

```markdown
The image below is from [Google][googleweb]

![Google's logo][logo]

[googleweb]: http://www.google.com
[logo]: https://www.google.com/images/srpr/logo3w.png
```

Сгруппировав сноски в конце файла, вы можете легко находить, редактировать и повторно использовать URL-адреса ссылок и изображений.
